r/golang on Reddit: Microservices with clean architecture
https://www.reddit.com/r/golang/comments/zbap13/microservices_with_clean_architecture/
You can achieve CLEAN with almost any folder structure. The most important thing about CLEAN is the conceptual layering and how those layers interact with each other, but your folder structure does not have to map nicely to each layer. Get that out of your head and just follow the principles. For microservices, I like to keep it very simple. Don't overstructure preemptively. Create a folder for the API layer, a folder for config and a folder for each entity. Keep the rest of the code on the top level, like the app, etc. This is what I call the "entity-based" structure. You achieve CLEAN here by saying that entities can't import any other package except for other entities, API can import entities but not app, and app can import everything. In other services where I know I will only have one entity but many different "adapters", I will make the structure "adapter-based", so each folder represents one adapter and the entity code lives under the root level or under "internal". root level can't import from adapters or app. Adapters are not allowed to import each other but they can import from the root level. App can import everything. But please, for the love of god, don't call your folders something like "infrastructure", "entities", "usecase". More on reddit.com
